# Implementation Plan: Forge CLI — Phase 1 (Plugin System)

Based on gap analysis vs Hermes CLI & OpenClaw CLI.

## Goal

Add a plugin/extension system to Forge CLI that allows third-party code to inject commands, modify TUI views, and hook into CLI lifecycle — modeled after OpenClaw's extension pattern (simpler than Hermes' 4-layer).

## Current Context

- Repo: `JonasAbde/hermes-forge-cli` @ main (v1.1.0)
- 24 commands registered in `src/index.ts` via Commander
- Existing `plugin` command in `src/commands/plugin.ts` — stubs for install/list/remove
- Existing `src/lib/pluginManager.ts` — simple npm-based plugin loader
- No extension directory, no manifest convention, no hooks

## Design Decisions

**Model:** OpenClaw extensions (simpler than Hermes)
- Extensions live in `~/.forge/extensions/` (user) and `.forge/extensions/` (project)
- Each extension is a directory with `manifest.yaml` + code
- Manifest specifies: name, version, commands, hooks, dependencies
- No pip/npm auto-install — user clones or copies extensions

**Why not Hermes' model?**
- Hermes' 4-layer discovery is overkill for a DevOps CLI
- pip entry-points make sense for Python, not Node/TypeScript
- OpenClaw's flat extension directory is easier to reason about

## Step-by-Step Plan

### Step 1: Extension Manifest Schema

**File:** `src/types.ts` — add types
```typescript
interface ExtensionManifest {
  name: string;
  version: string;
  description?: string;
  commands?: ExtensionCommand[];
  hooks?: {
    onInit?: string;        // path to module
    onCommand?: string;     // path to module
    onBeforeExit?: string;  // path to module
  };
  dependencies?: Record<string, string>;
}

interface ExtensionCommand {
  name: string;           // subcommand name (e.g., "myplugin:deploy")
  description: string;
  module: string;         // path relative to extension root
  aliases?: string[];
}
```

### Step 2: Extension Loader

**File:** `src/lib/extensionManager.ts` — NEW (replaces/receives pluginManager)

Responsibility:
- Scan `~/.forge/extensions/` and `.forge/extensions/`
- Parse `manifest.yaml` from each extension dir
- Validate schema (required fields, no name conflicts)
- Return list of valid extensions

Key function: `loadExtensions() → ExtensionManifest[]`

### Step 3: Extension Command Injection

**File:** `src/lib/extensionManager.ts` — add `injectCommands(program: Command)`

Logic:
- For each extension with commands[]
- Register each command as `program.command('ext-<name>:<cmd>')`
- Lazy-import the command module when triggered
- Extensions cannot override built-in commands (name collision = warn + skip)

### Step 4: Extension Hook System

**File:** `src/lib/extensionManager.ts` — add hook dispatch

Hooks:
- `onInit(program)` — called after all commands registered
- `onBeforeExit()` — called before process exit
- `onCommand(commandName, args)` — middleware pattern

Implementation:
- Simple event emitter pattern
- Hooks run sequentially, errors logged but don't crash CLI

### Step 5: Extension Management Commands

**File:** `src/commands/plugin.ts` — UPDATE

Commands:
- `forge plugin list` — list installed extensions, their commands, status
- `forge plugin install <path|git-url>` — clone/copy to ~/.forge/extensions/
- `forge plugin remove <name>` — delete extension directory
- `forge plugin enable/disable <name>` — toggle without deleting

### Step 6: .forgeignore for extensions

**File:** `.forgeignore` — NEW
```
node_modules/
.env
*.log
```

## Extension Manifest Example

```yaml
# ~/.forge/extensions/forge-deploy/manifest.yaml
name: forge-deploy
version: 1.0.0
description: "Advanced deployment tools for Forge"

commands:
  - name: deploy:rollback
    description: "Roll back to a previous deployment"
    module: "commands/rollback.js"
    aliases: [deploy:rb]

  - name: deploy:status
    description: "Show deployment history"
    module: "commands/status.js"

hooks:
  onInit: "hooks/init.js"
  onBeforeExit: "hooks/cleanup.js"
```

## Risks & Tradeoffs

- **Security:** Extensions run with full CLI privileges. No sandboxing in v1.
  - Mitigation: Warn on install, `forge plugin list` shows installed count, `forge doctor` validates manifests
- **Name collisions:** Extension commands prefixed with `ext-<name>:` to avoid shadowing built-ins
- **Performance:** Scanning extensions adds ~5-10ms to startup. Acceptable.
- **Dependencies:** Extensions use `require()` from the CLI's node_modules — dependency conflicts possible
  - Mitigation: Document that extensions should pin their deps
