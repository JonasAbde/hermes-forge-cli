# Forge CLI v2.0 — Vision Document

> **"The terminal is the forge. The CLI is the hammer. The platform is the anvil."**

---

## 1. Why Forge CLI v2.0 Must Exist

Forge CLI v1.x is a **good DevOps tool**. It checks health, manages packs, runs dev servers, and talks to the remote API. It solves real problems.

But it is not yet **the primary interface** for the Forge platform. It is a sidekick to the web UI — helpful, but optional.

**v2.0 changes that.** The vision is a CLI so deeply integrated that developers *choose* it over the web UI for everything except card art browsing and social features. A CLI that:

- **Feels like playing the game**, not administering it
- **Brings the game into the terminal** — leaderboards, pack openings, fusion
- **Is smarter than you** — natural language, proactive suggestions, AI spawning
- **Integrates horizontally** — Telegram bot, MCP server, GitHub Actions, marketplace
- **Gets better the more you use it** — XP, achievements, unlocks

This document lays out the concrete features, architectural shifts, and philosophical foundations for that transformation.

## 2. The Four Pillars of Innovation

### Pillar I: The CLI as a Living Interface ™

**Innovation:** Most CLIs are *read-only* windows into a system. Forge CLI v2.0 is a *bidirectional living interface* — it not only reads the platform state, it *contributes* to it, *earns* from it, and *evolves* with it.

**Concrete features:**

| Feature | Rationale |
|---------|-----------|
| **`forge watch`** — persistent live view that streams platform events (pack opened, agent fused, leaderboard change) to the terminal in real-time | Turns the terminal into a live arena feed; replaces refreshing the web UI |
| **`forge pack open [bundle]`** — animated pack opening in terminal (spinning cards, rarity reveal, sound via terminal bell) | The most satisfying feature of the web UI, now in your terminal. Uses box-drawing, ascii animations, and color gradients. Runs offline via deterministic reveal |
| **`forge arena`** — interactive TUI that shows your loadout, active agents, ongoing battles, and lets you queue actions | Replaces the dashboard entirely; the terminal becomes the command center |
| **`forge notification stream`** — desktop notifications (via `node-notifier`) when platform events match your watchlist | Stay connected without keeping a browser tab open |
| **`forge quote`** — pulls a random agent quote from the catalog to your terminal; each pack has unique flavor text | Delight; makes the CLI feel alive and characterful |

**Why this is innovative:** It inverts the relationship. Instead of the CLI being a *tool you use to interact with the platform*, the platform *comes to you* through the CLI. The terminal becomes a node in the platform's event mesh, not just an HTTP client.

---

### Pillar II: Gamified Development — XP, Achievements, and CLI Identity

**Innovation:** Your CLI usage *is part of the game*. Every `forge` command earns XP for your account. Achievements unlock terminal themes, custom ASCII art, and CLI-only cosmetics.

**Concrete features:**

| Feature | Rationale |
|---------|-----------|
| **`forge me`** — shows your CLI profile: level, XP, achievements, rank on leaderboard | Identity and progression visible in the tool itself |
| **`forge achievements`** — browse CLI achievements: "First Pack Sync", "100 Health Checks", "TUI Marathon (10h in TUI)", "Fusion Master", "Marketplace Mogul" | Turns CLI usage into a meta-game with unlockable rewards |
| **`forge leaderboard`** — global leaderboard of CLI users by XP, commands run, packs synced, deployments created | Social competition drives engagement |
| **`forge skin apply <theme>`** — unlockable terminal themes tied to achievements (e.g., "Void Walker" for 50 health checks, "Pack Rat" for owning 500 packs) | Cosmetic rewards that make achievements visible to others |
| **`forge streak`** — shows your daily usage streak; streaks multiply XP gain | Daily active usage is the holy grail of developer tools |

**Earning model:**

```
forge pack list           → +5 XP
forge pack sync           → +25 XP
forge deploy create       → +50 XP
forge arena (15 min TUI)  → +100 XP
forge marketplace trade    → +75 XP
forge achievement unlock   → bonus +500 XP
Daily first command        → +10 XP (streak starts)
7-day streak               → 2x multiplier on all XP
```

**Why this is innovative:** No developer CLI has done this. Not `gh`, not `aws`, not `kubectl`. By making CLI usage part of the game's progression system, Forge CLI becomes *sticky* — developers keep using it because their identity is invested in it. It transforms a utilitarian tool into a platform experience.

### Pillar III: AI-Native — Natural Language, Proactive Agents, MCP as the Bridge

**Innovation:** The CLI doesn't just *have* AI features — it *is* an AI interface. Natural language commands, proactive agent spawning, and bidirectional MCP integration make the CLI the smartest way to interact with Forge.

**Concrete features:**

| Feature | Rationale |
|---------|-----------|
| **`forge ask "deploy my strongest fire agent"`** — natural language command parsing; uses local LLM or calls forge.tekup.dk's AI to interpret intent and execute multi-step workflows | Lowers the barrier to complex operations; "just say it" |
| **`forge agent spawn --pack <id>`** — spawns an AI agent from a pack directly into your Hermes Agent runtime from the CLI | The core action of the platform, now CLI-native |
| **`forge suggest`** — proactive suggestions based on your state: "You have 3 unopened packs. You could fuse the 'Pyromancer' and 'Inferno Elemental' for a 40% stat boost." | The CLI becomes a strategic advisor, not just a command executor |
| **`forge plan "grow my collection to 50 unique agents"`** — generates a multi-step action plan using the Forge AI | Long-term goal planning that executes across commands |
| **`forge mcp start`** — starts the Forge MCP server as a child process, exposing CLI commands as MCP tools for any MCP client (Cursor, Claude Desktop, etc.) | The CLI becomes both consumer AND provider of MCP — a bridge between platforms |
| **`forge mcp call <tool> <args>`** — call any MCP tool (from any server) directly from the CLI | The CLI becomes the universal MCP client |

**MCP bidirectional architecture:**

```
┌─────────────────────────────────────────────────┐
│                  Forge CLI v2.0                  │
│                                                   │
│  ┌──────────────┐    ┌──────────────────────┐    │
│  │  MCP Consumer │    │    MCP Provider       │    │
│  │  (MCP client)  │    │  (exposes CLI as MCP) │    │
│  └──────┬───────┘    └──────────┬───────────┘    │
│         │                       │                  │
│         ▼                       ▼                  │
│  ┌──────────────────────────────────────────┐    │
│  │         forge.tekup.dk API               │    │
│  └──────────────────────────────────────────┘    │
│                                                   │
│         ◄── MCP protocol ──►                      │
│         (Cursor, Claude, VS Code)                 │
└─────────────────────────────────────────────────┘
```

**Why this is innovative:** Most CLIs are passive — you type, they respond. Forge CLI v2.0 is *proactive*, *conversational*, and *context-aware*. It uses MCP not as an afterthought but as the fabric that connects the CLI to every other tool in the ecosystem. The CLI becomes the center of a tool network.

### Pillar IV: Ecosystem Hub — Marketplace, Telegram, CI/CD, Plugins

**Innovation:** The CLI is not just a client for forge.tekup.dk — it is the *control plane* for the entire Hermes Forge ecosystem, including Telegram bots, GitHub Actions, and third-party plugins.

**Concrete features:**

| Feature | Rationale |
|---------|-----------|
| **`forge marketplace list`** — browse the marketplace from terminal; filter by rarity, price, agent type | Full marketplace access without leaving the terminal |
| **`forge marketplace buy <listing-id>`** — execute trades from CLI; requires 2FA confirmation | Fast trading for power users |
| **`forge market watch <agent-name>`** — sets a price alert: notifies when a specific agent drops below a threshold | Proactive market intelligence |
| **`forge telegram link`** — link your Telegram account; allows trading and deployment commands from Telegram via the CLI's MCP bridge | Cross-platform identity and action |
| **`forge ci status`** — shows status of the latest GitHub Actions run for the platform repo | Devs live in CI; bring CI to the terminal |
| **`forge deploy --watch`** — deploy and stream build logs in real-time | Deployments are the most anxious moment; live logs reduce anxiety |
| **`forge plugin publish`** — package and publish a CLI plugin to the Forge Plugin Registry | Ecosystem growth through community contributions |
| **`forge doc search "fusion rates"`** — full-text search of Forge Docs (VitePress) from the terminal | Documentation discoverability without leaving the workflow |
| **`forge remote exec "<sql>"`** — run read-only SQL queries against the platform database (with proper auth and auditing) | Power users need data access; this is safer than sharing DB dumps |

**Plugin ecosystem maturity model:**

| Phase | Capability | Example |
|-------|-----------|---------|
| v2.0.0 | Extension manifest + static command injection | Custom deploy scripts |
| v2.1.0 | Lifecycle hooks + TUI widget injection | Custom dashboard widgets |
| v2.2.0 | Event subscriptions + MCP tool exposure | Telegram bot plugins |
| v2.3.0 | Plugin marketplace with ratings and XP | Community publishing |

**Why this is innovative:** The CLI becomes a *hub* — a single place where marketplace, Telegram, CI/CD, documentation, and plugin management converge. Instead of switching between 5 different tools and tabs, developers stay in the terminal. The terminal is the platform.

## 4. Technical Innovations

### 4.1 The Event Mesh

Forge CLI v2.0 runs a persistent WebSocket connection to `forge.tekup.dk/api/events` when in `watch` mode. Events are structured as:

```json
{
  "type": "pack.opened",
  "data": { "userId": "...", "packId": "...", "rarity": "legendary" },
  "timestamp": "2026-05-13T10:00:00Z"
}
```

The CLI processes these events with a composable pipeline:
- **Router** — dispatches events to registered handlers (watch, notification, streak update)
- **Filter** — user-configurable event filters via `forge watch --filter "rarity=legendary"`
- **Renderer** — renders events to terminal with animation, sound, or notification

### 4.2 The XP and Achievement Engine

The CLI maintains a local SQLite database (`~/.forge/profile.db`) that caches:
- XP balance and level
- Achievement progress
- Streak data
- Command history (last 1000 commands)

This is synced to the platform via API calls. The local cache allows:
- **Offline XP accrual** — command usage when disconnected still counts; synced on reconnection
- **Instant feedback** — no network latency for XP notifications
- **Privacy** — command history never leaves the local machine unless user opts-in

### 4.3 24 CLI Commandments (Manifesto)

```
═══════════════════════════════════════════════════════
  THE 24 CLI COMMANDMENTS
═══════════════════════════════════════════════════════

  I.   THE CLI IS THE PRIMARY INTERFACE — not a sidekick to the web UI.
       Every web feature must have a CLI equivalent.

  II.  SPEED IS SANCTITY — commands must feel instant (<100ms for local,
       <500ms for API calls). No command should take longer than brewing coffee.

  III. BEAUTY IS NOT OPTIONAL — ASCII art, color gradients, progress bars,
       and thoughtful typography make the CLI a joy, not a chore.

  IV.  THE TERMINAL IS THE ARENA — pack openings, fusion animations, and
       leaderboards belong in the terminal, not just the browser.

  V.   EVERY COMMAND EARNS XP — usage is participation. The more you CLI,
       the more powerful you become.

  VI.  AI IS THE DEFAULT — natural language parsing, proactive suggestions,
       and autonomous execution are not features, they're expected.

  VII. MCP IS THE BRIDGE — the CLI is both consumer and provider of models,
       context, and protocols. It connects everything.

  VIII. OFFLINE FIRST — the CLI must work without network for local operations
       (validation, preview, simulation). Sync is async.

  IX.  PLUGINS ARE CITIZENS — the plugin API is as important as the built-in
       commands. Community extends the forge.

  X.   CONFIG IS CODE — forge.config.yaml is executable documentation.
       Every setting has a command-line equivalent.

  XI.  HELP IS NOT AN AFTERTHOUGHT — `forge help`, `forge <cmd> --help`,
       and `forge doc` must be exceptional. Documentation lives in the CLI.

  XII. COMPOSITION OVER MONOLITH — commands compose via pipes, flags, and
       chaining. `forge pack list | forge market watch` should work.

  XIII. TRANSPARENCY IS TRUST — the CLI shows you exactly what it's doing,
       what API calls it makes, and how long they take. `--verbose` is honest.

  XIV.  DELIGHT IS DESIGN — random agent quotes, easter eggs, celebrations
       on achievements. The CLI should make you smile.

  XV.   EVERY VIEW HAS A TUI — if it can be displayed, it can be a dashboard
       widget. `forge tui` is the ultimate interface.

  XVI.  AUTH IS ZERO-FRICTION — magic link from the terminal. No API keys
       to manage. Login once, use everywhere.

  XVII. COMMANDS ARE MEMORABLE — short, consistent, hierarchical.
       `forge pack open` not `forge open-pack`. `forge market buy` not
       `forge marketplace-purchase`.

  XVIII. FAILURES ARE BEAUTIFUL — error messages are explanations with
        suggested fixes. Stack traces are the last resort.

  XIX.  THE CLI TELEGRAPHS INTENT — `--dry-run` and `--simulate` on every
        mutating command. Preview before you commit.

  XX.   VERSIONING IS CONTRACT — semantic versioning with clear breaking
        change policies. `forge upgrade` handles migrations.

  XXI.  TELEMETRY IS OPT-IN AND ANONYMOUS — usage data helps improve the
        CLI. But it's always opt-in, always anonymized, always deletable.

  XXII. THE CLI IS A PLATFORM, NOT A TOOL — extensions, themes, hooks,
        and MCP tools make it extensible by design.

  XXIII. ACCESSIBILITY IS NOT OPTIONAL — 256-color support, high-contrast
         themes, screen reader compatible output. The forge is for everyone.

  XXIV.  THE CLI HAS PERSONALITY — it's called "Forge CLI" and it has a
         voice. Slightly dramatic, deeply competent, occasionally hilarious.

═══════════════════════════════════════════════════════
```

## 5. The "Why Not Web?" Matrix

Every feature must answer: **Why would a developer choose the CLI over the web UI?**

| Feature | CLI Advantage | Web Advantage |
|---------|--------------|---------------|
| Pack opening | Animated ASCII + instant feedback + offline | Card art, animations, sound |
| Marketplace trading | Keyboard speed, watchlists, alerts | Visual browsing, card images |
| Deployment | Live logs, CI integration, MCP exposure | Deployment history UI |
| Collection browsing | Sort/filter with grep power, export | Visual card grid |
| Fusion | Simulation, probability analysis | Visual animation, card art |
| Leaderboard | Real-time refresh, filtering | Social features, profiles |
| Agent chat | Wait... this should be CLI-native too | Full chat UI with streaming |

**The winning strategy:** Make the CLI the *power tool* and the web the *showroom*. Power users live in the CLI. Casual users browse the web. The two are synchronized — actions in one appear in the other immediately via the event mesh.

## 7. Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| **Scope creep** — 24 commandments is a lot | Phase delivery; v2.0.0 ships Pillar I + III core; v2.1.0 adds gamification; v2.2.0 adds plugins |
| **Performance** — WebSocket + TUI + AI is heavy | Profile aggressively; WASM for critical paths; optional features via config |
| **API dependency** — CLI is useless if forge.tekup.dk is down | Offline-first design; local simulation mode; graceful degradation |
| **User overwhelm** — too many features | Progressive discovery; "Did you know?" tips; contextual help |
| **Maintenance burden** — 24+ commands is a lot to maintain | Strong test coverage (target: 80%+); plugin architecture shifts maintenance to community |
